# 학습한 규칙 (smartstore-addnew 프로젝트)

## 0-A. 워크플로 시작 전 — 국내 시세 먼저 확인 (필수)

사진+가격 데이터 받으면 **산출물 4종 만들기 전에** 다음을 먼저 수행한다:

1. **쿠팡 검색** — 같은/유사 SKU가 국내 정식 유통(특히 로켓배송)으로 팔리는지 확인
2. **네이버 쇼핑 검색** — 동일 SKU 가격대 / 셀러 수 / 최저가 확인 (G마켓·11번가·SSG·롯데홈쇼핑 포함)
3. **시세 보고:**
   - 동일/유사 SKU 채널별 최저가
   - 내 책정가(상품가+배송비 실수령 기준)와 차이
   - "가격 경쟁력 있음 / 박빙 / 가격으로 안 됨" 한 줄 판단
4. 사용자에게 **A) 진행 B) 가격 조정 C) 접고 다른 SKU** 선택지 제시
5. **명시적 GO 받은 뒤에만** 등록정보.md / detail.html / product_info.json 산출물 생성 시작

**이유:** 박한 마진($2 등) 책정가가 한국 정식 유통(쿠팡 로켓·홈쇼핑 폭탄가) 대비 경쟁력 없으면 등록·발송 노력 매몰. **"가격으로 안 되면 걍 접는다"** 원칙 (2026-05-17 Snuggle 드라이어시트 케이스에서 학습 — 책정가 ₩13,900+배송 ₩18,900 vs 쿠팡 로켓 ₩20,000, 롯데홈쇼핑 ₩12,900 무료배송).

**예외 (스킵 트리거):**
- 사용자가 "시세 확인 스킵하고 바로 진행"이라고 명시한 경우.
- 사용자가 **"이건 한국에 안 파는 제품"**, **"국내에 없다"**, **"국내 미출시"**, **"한국 미출시"** 등으로 명시한 경우 — 비교할 국내 시세 자체가 없으므로 §0-A를 스킵하고 책정한 가격 패턴 그대로 진행. (2026-05-17 Kellogg's Frosted Flakes Cookies & Crème Milkshake 케이스에서 학습 — Cookies & Crème Milkshake 변형은 한국 미출시 한정판이라 쿠팡·네이버 시세 비교 불가.)

> 룰 적용: "한국에 안 판다" 류 멘트가 메시지에 있으면 시세 확인 자동 스킵하되, 보고에는 **"국내 미유통 — §0-A 스킵 적용"**이라고 1줄 명시.

## 0-B. 폰에서 빠르게 보낼 때 (Dispatch / 일반 메시지 공통)

사진과 함께 아래 형식 중 하나로 보내면 추가 질문 없이 바로 워크플로 진입.

**블록 형식:**
```
[사진]
스토어: Loblaws       ← 현재 보고 있는 매장 (공식 제품명 추출 소스)
가격: $6.29 (세전)
세금: 0%
마진: 15%   (또는 $5)
배송: 2개당 15000
모드: single
엔진: agents   (선택 — 있으면 서브에이전트 모델)
```

**스토어 = 현재 보고 있는 매장 → 모델명 추출 (2026-05-31 추가):** `스토어` 필드는 단순 출처
라벨이 아니라 **공식 제품명을 끌어올 소스 매장**이다. 그 매장에서 official 영어 풀네임을 추출해
네이버 **모델명** 필드를 채운다(그룹은 옵션별). 크롤 가능 환경(URL·Cowork dispatch)은 소스에서
직접 추출, 크롤 차단(사진만)이면 추측 단정 말고 사용자에게 확인. 상품명에는 국가명(캐나다/미국)을
넣지 않는다 — 네이버 원산지 필드가 `[원산지:캐나다]` 자동 표시(중복 방지). (CLAUDE.md §5,
[[feedback_model_name_from_store]])

**한 줄 형식:**
```
[사진]
Loblaws / $6.29 / 0% / 15% / 2개당 15000 / single
```
서브에이전트 모델: 끝에 `/ agents`(또는 `에이전트`) 토큰 추가.

빠진 항목만 묻고, 모드 누락은 single 기본.

**실행 엔진 (2026-05-30 추가):** 입력에 `엔진: agents` / `agents` / `에이전트` 플래그가 있으면
서브에이전트 모델(`.claude/commands/register-agents.md` 오케스트레이션 — 시세조사·추출·작성을
`market-researcher`·`product-extractor`·`listing-writer` 에 위임, 가격계산·GO게이트·저장·Slack 은
메인 소유)로 실행한다. 플래그 없으면 기존 기본 워크플로. dispatch 는 슬래시 커맨드(`/register-agents`)를
확장하지 못하므로 이 플래그가 dispatch 의 유일한 서브에이전트 진입점이다 (CLAUDE.md §1 엔진 플래그 규칙).

## 0-C. 라인업/그룹 형제 가격 비교 — 매입처가 같아야 유효 (2026-06-14)

그룹상품·라인업 형제 판매가를 신규 SKU 가격 책정 기준으로 쓸 때 **각 형제의 매입처(코스트코·월마트 등)가 신규 SKU 와 같은지 먼저 확인**한다. 매입처가 다르면 원가 baseline 이 달라(코스트코 회원가/대용량 < 월마트 소매가) 판매가를 직접 비교하면 안 된다. 매입처 다른 형제보다 신규가가 높아도 **자동으로 "과하다" 판정 금지** — 원가(손익분기) 기준으로 책정하고 근거 명시.

> 2026-06-14 다우니 언스토퍼블 파라다이스 853g(월마트 소싱, 원가 $24.97, 손익분기 ₩33,120) → ₩36,900. 그룹 910g 형제(씨사이드/프레시 ₩29,900)보다 비싸 "과하다"고 플래그했더니 사용자: "그 둘은 코스트코, 이건 월마트라 비교 불가." [[feedback_source_store_price_comparison]]

## 1. 네이버 스마트스토어 수수료 항상 6.6% (gross-up)

스마트스토어 판매가는 항상 네이버 수수료 6.6%를 차감 후 마진이 남도록 계산.

**계산식:**
```
sell_pre_fee = cost_original + markup            (CAD or USD 단위)
sell_original = sell_pre_fee / (1 - 0.066)       (수수료 gross-up)
sell_krw = ceil_10(sell_original × exchange_rate)
```

**예시** — Chosen Foods Avocado Mayo (CAD $12.99 + $5 마진, 환율 1,083):
- 12.99 + 5 = 17.99
- 17.99 / 0.934 = 19.26 CAD
- 19.26 × 1083 = 20,859.85 → ceil_10 → **₩20,860**

**이유:** "$5 남기고 싶다"고 할 때 그 $5는 네이버 수수료 차감 후의 순이익이어야 의미가 있음. 수수료 적용 안 하면 실제 수익이 6.6% 깎임.

**예외:** 사용자가 명시적으로 "수수료 빼" / "수수료 0%"라고 한 경우만 미적용.

`product_info.json`에는 `platform_fee_rate: 0.066`, `sell_pre_fee`, `sell_original` 모두 기록.

## 2. 브랜드명 한글 표기는 네이버 검색량 기준

외래 브랜드명을 한글로 옮길 때 **음운적 정확성보다 네이버 검색량이 우선**. 셀러 입장에서 검색 노출이 매출 직결이라 표준 표기보다 실제 소비자가 치는 단어 따라감.

**확정된 표기:**
- Chosen Foods → **초슨푸드** (NOT 초즌푸드)
- President's Choice → **PC 프레지던트초이스** (병기 — 패키지 약자 PC + 정식 한글 브랜드. **어순은 `PC 프레지던트초이스`**가 어감 우선, 2026-06-26 사용자 정정. 이전 `프레지던트초이스 PC`에서 PC를 앞으로). PC 단독은 검색 모호(컴퓨터)라 풀네임과 병기. 동의어 중복 금지(§5)의 예외 — 사용자 명시 병기.

## 3. 캐나다 세금 규칙 (항상 사용자 확인, 가격은 세전)

**핵심 원칙:**
1. 사용자가 주는 가격은 **세전(pre-tax)**. 매장 표시가 그대로.
2. 세금은 **매번 물어본다.** 카테고리만 보고 자동 적용 금지.

**참고 세율** (질문할 때 후보로 제시 — 확정 아님, 사용자 판단 우선):

| 카테고리 | 보통 세율 |
| --- | --- |
| 일반 상품 (가공식품·캔디·스낵·음료·생활용품 등) | HST 13% |
| 소스 (마요네즈·머스타드·케첩·드레싱 등) | 0% |
| 커피류 (캡슐·원두·분쇄) | 0% |
| 쿠키 | 0% |
| 키즈 의류 (kids size) | GST 5% |
| 신발 | GST 5% |

> 키즈 의류·신발은 0%가 아니라 **PST 면제 + GST 5%만 적용**이라 5%.

**계산식** (세전 가격 기준):
```
cost_with_tax = cost_pre_tax × (1 + tax_rate)      (0%/5%/13%)
sell_pre_fee = cost_with_tax + markup              (마진 적용)
sell_original = sell_pre_fee / (1 - 0.066)         (네이버 수수료 gross-up)
sell_krw = ceil_10(sell_original × exchange_rate)
```

## 4. 상세페이지에 검증 안 된 활용 팁/레시피 섹션 만들지 말기

스마트스토어 상세페이지 만들 때 다음 종류의 섹션은 **자동 생성하지 않음**:
- "활용 팁" / "사용 팁" / "이렇게 드세요"
- "추천 레시피" / "조리 예시"
- 사용 시나리오, 페어링 제안 등 LLM 추측 기반 콘텐츠

**유지하는 섹션** (브랜드/스토어 페이지에서 가져온 검증된 정보만):
- 상품 설명 (스토어 페이지에서 추출한 description)
- 상품 포인트 / 특징 (브랜드가 명시한 features, certifications)
- 제품 사양 (브랜드/용량/원산지/보관방법)
- 원재료 / 알레르기 (라벨 또는 브랜드 페이지 출처)
- 배송 안내 (사용자 규칙)

**이유:** 리셀러가 상세페이지를 고객용으로 그대로 올림. 검증 안 된 활용 팁은 부정확하거나 오해 소지 → 컴플레인/반품 리스크. 프로젝트 규칙 §9 데이터 진실성 원칙과 직결.

**예외:** 사용자가 명시적으로 "활용 팁도 넣어줘" / "레시피 아이디어 추가해" 라고 요청한 경우.

**🔑 한국 고객과 무관한 인증/표기는 상세페이지에서 제외 (2026-06-26):** **코셔(Kosher·COR·Pareve)**, 할랄 등 한국 소비자에게 의미 없는 종교/지역 인증은 상세페이지 본문에 넣지 않는다(사용자: "한국 사람 중에 코셔 하는 사람 없으니"). product_info `verified_claims` 에는 사실 기록으로 남겨도 되지만 고객용 detail.html 에는 미표기. (Non-GMO·유기농·비건·무첨가 등 한국에서도 소구되는 인증은 유지.)

## 5. 원가에 HST 포함된 경우 — HST 가산 스킵

평소 룰(§3)은 "사용자가 주는 가격은 세전"이지만, 사용자가 **명시적으로 "HST 포함" / "tax-in" / "세금 포함"** 이라고 한 경우에는 HST를 다시 곱하지 않는다.

**계산식 (HST 포함 원가):**
```
cost_with_tax = cost_original                       (그대로, 가산 X)
sell_pre_fee = cost_with_tax + markup
sell_original = sell_pre_fee / (1 - 0.066)          (네이버 수수료 gross-up은 동일)
sell_krw = round_100(sell_original × exchange_rate)
```

**예시** — Aveeno Baby Lotion 532mL (2026-05-16):
- 원가 16.99 CAD (**HST 포함**, Costco 매대 표시가)
- 16.99 + 3 (마진) = 19.99 CAD
- 19.99 / 0.934 = 21.4026 CAD
- × 1,070 = 22,901 → round_100 → **₩22,900**
- 검산: 22,900 ÷ 1,070 × 0.934 − 16.99 = **2.9993 CAD 마진** ✅

**이유:** 일부 매장(Costco 회원가, Wholesale 등)은 매대 표시가 세금 포함. 평소 룰대로 HST를 다시 곱하면 cost가 13%만큼 부풀려져서 한국 판매가가 비싸지고 실제 마진은 안 늘어남.

**기록:** `product_info.json`의 `tax_label`은 "HST 13% (included in cost)", `tax_added_to_cost: false` 로 표기해서 추후 추적 가능하게.

> 명시 안 되어 있으면 평소대로 §3 적용 (세전 가정 + 사용자에게 세율 확인).

## 6. 목표가 매칭 산식 (cost-plus 스킵)

사용자가 **"가격 N$로 맞추고/매칭/맞춰"** / **"네이버에 N(원)에 올려"** / **"다른 X 제품과 가격대 맞춰"** 라고 명시하면 원가/마진 계산을 **스킵**하고 N을 **목표가**로 사용한다.

### 6-1. 단위 해석 — **반드시 먼저 확정** (중요)

"N$" / "N불" / "N원" 표현은 단위가 애매할 수 있다. 다음 순서로 해석한다:

1. **"N원" / "N(콤마/숫자만)" / "원"이 붙은 표현** → **KRW** (네이버 listed price, post-fee, gross-up·환산 없음)
2. **"N$" / "N불"** + **한국 판매가 맥락**(네이버 노출가, 라인업 매칭 등) → **1차 해석 ₩N,000 (천원 단위)** 로 가정. 한 자릿수 $는 한국 일상 표현에서 "N천원"으로 자주 쓰임.
3. **"N CAD" / "N 캐나다 달러"** 또는 원가/마진 맥락이면 → **CAD**, gross-up·환산 적용 (아래 6-3).
4. 1·2·3 어느 쪽인지 명확히 판단되지 않으면 **반드시 사용자에게 확인** 후 진행 (잘못 책정하면 마진·전환율 양쪽 다 깨짐).

> 2026-05-17 Kellogg's Frosted Flakes 케이스에서 학습: "가격 8$로 맞춰" → 처음에 8 CAD로 해석해 ₩9,200 책정 → 사용자 정정 "아니 그냥 네이버에 8000원에 올려" → ₩8,000. 다음부터는 한 자릿수 $ + 한국 판매가 맥락이면 ₩N,000 으로 1차 가정.

### 6-2. KRW 직접 지정 (₩N 또는 ₩N,000)

사용자가 KRW로 직접 목표가를 지정한 경우 — gross-up·환산 없이 **그대로 listed price**.

**계산식 (KRW 직접):**
```
sell_price_krw = N (KRW, post-fee)                  (사용자 지정 그대로)
실 수령가 = N × 0.934                                  (수수료 차감 후)
실 수령가_CAD = (N × 0.934) ÷ FX                       (참고용 환산)
```

**예시** — Kellogg's Frosted Flakes Cookies & Crème Milkshake 333g (2026-05-17, FX 1,070):
- 목표가 ₩8,000 (사용자 정정 후)
- 8,000 × 0.934 = **₩7,472 (실수령)**
- 7,472 ÷ 1,070 = **6.98 CAD pre-fee** (참고)
- **최종 판매가 ₩8,000**

### 6-3. CAD 목표가 (gross-up + 환산)

**계산식 (CAD 목표가):**
```
target_sell_pre_fee = N CAD                           (사용자 지정 목표가)
sell_original = target_sell_pre_fee / (1 - 0.066)     (수수료 gross-up)
sell_krw = round_100(sell_original × exchange_rate)
```

**이유:** "다른 라인업과 가격대 맞추기" 같은 요구는 마진보다 **가격 포지셔닝**이 우선. 원가에서 시작하면 라인업 가격대가 어긋남. CAD 단위로 명시한 경우 gross-up은 적용해야 라인업 실 수령가 기준 같은 마진폭 유지.

**기록:** `product_info.json`에 `pricing_mode: "target_match"`, `target_currency: "KRW"` 또는 `"CAD"`, `target_value: N`, `cost_original`과 `markup`은 참고용/없으면 null.

**예외:** 사용자가 "gross-up 빼" / "수수료 미적용"이라고 한 경우.

### 6-4. 목표가 직접 지정 + 원가 있음 → 실 마진 **역산** (기록·검산 필수)

§6-2/6-3은 목표가를 그대로 listed price로 쓰는 규칙. 여기에 더해 — 사용자가 **목표 판매가를 직접 지정**하면서 **원가(±HST)도 함께 준 경우**, cost-plus 산식은 스킵하되 **실효 마진을 역산해서 등록정보·product_info에 반드시 기록**한다. (사용자가 "마진 역산해서 기록" 류로 명시하면 항상 적용.)

**역산 산식 (KRW 목표가 기준):**
```
after_fee_krw   = target_krw × 0.934                  (네이버 수수료 차감 실수령)
realized_cad    = after_fee_krw ÷ FX                  (CAD 환산)
cost_with_tax   = cost_original × (1 + tax_rate)      (0/5/13% — §3·§5 규칙대로)
실 마진(CAD)    = realized_cad − cost_with_tax
마진율          = 실 마진 ÷ realized_cad
```

**예시** — Tim Hortons ACE 팀빗 홀더 (2026-05-20, 목표가 ₩29,900, 원가 16.99 CAD, HST 13%, FX 1,090):
- 29,900 × 0.934 = 27,926.6 KRW (실수령)
- 27,926.6 ÷ 1,090 = 25.6207 CAD
- 16.99 × 1.13 = 19.1987 CAD
- 25.6207 − 19.1987 = **6.42 CAD 실 마진** (마진율 = 6.42 ÷ 25.62 ≈ 25%)

**기록 위치:** `product_info.json` → `pricing_mode: "target_match"`, `verification.method: "reverse_margin"`, `markup`에 역산값 + "역산" 표기, `verification.margin_comment`에 **마진 합리성 한 줄 코멘트**. `등록정보.md`에도 역산 블록 + 코멘트.

**마진 합리성 코멘트 기준:** 일반 유통 식품은 마진대 $2~5가 정상 범위. 역산 마진이 이를 초과하면 **무조건 과하다고 보지 말 것** — 한국 미출시·한정·콜라보·컬렉터블/굿즈류는 국내 경쟁 SKU 부재로 가격 탄력성이 커서 $6~ 이상도 합리적. 코멘트에 "(품목 특성상 합리적 / 재검토 권장)" 중 하나를 1줄로 판단해 적는다.

> 2026-05-20 ACE 팀빗 홀더 케이스에서 학습 — "목표 판매가 직접 지정 → 마진 역산" 패턴이 반복되어 정식 룰화. §6-2(KRW 직접 지정)의 확장: 원가가 같이 주어지면 실수령에서 그치지 말고 실 마진까지 역산·기록한다.

## 7. 배송비 "수량당 N원 (몇 개당 그런 거 없이)" — 단순 곱셈 패턴

사용자가 **"배송비 N원 (몇 개당 그런 거 없이)"** / **"수량당 N원"** / **"개당 N원, 묶음 없음"** 으로 명시하면 **번들/묶음 룰 없이 수량 × N원** 단순 곱셈 적용.

**계산:**
```
shipping_total = quantity × N
```
- 1개 주문 → N원
- 2개 주문 → 2N원
- 3개 주문 → 3N원 (3개당 할인 없음)

**네이버 배송 설정:**
- 수량당 N원으로 설정 (묶음할인 X)
- "수량당 5,000원" / "수량당 15,000원" 같은 형태로 등록

**기존 패턴과 구분 (§7-기존):**

| 사용자 표현 | 해석 |
| --- | --- |
| "배송비 5,000원, 3개당 15,000원" | 개당 5,000원, 3개 살 때도 15,000원 (묶음 룰 동일 = 단순 곱셈이지만 묶음 명시) |
| "3개당 15,000원" | 묶음할인 (1·2개 = 5,000원/개, 3개부터 15,000원) — Haribo 1kg 케이스 |
| **"N원 (몇 개당 그런 거 없이)"** / "수량당 N원" | **수량별, 단순 곱셈** (1개=N, 2개=2N, 3개=3N…). 네이버 배송비유형 **수량별**, **부과수량(AP)=1** |
| **"그냥 flat N원 (개당 없이)"** / "균일 N원" / "수량 상관없이 N원" | **유료(균일), 수량 무관** (1개든 5개든 N원 한 번). 네이버 배송비유형 **유료**, **부과수량(AP)=blank(공란)** — 신규 (2026-06-20) |
| "무배 + 흡수 N CAD" | 네이버 무료배송 등록 + 상품가에 N CAD 흡수 (메모리 feedback_free_shipping_absorb) |
| "M개당 N원" | M개 묶음할인 (M 이하 = N원, M+1부터 추가 묶음당 N원). 네이버 **수량별**, **AP=M** |

>
> **🔑 "수량당 N (단순 곱셈)" vs "flat N (균일)" 구분 (2026-06-20 트위즐러 케이스):** 둘 다 "개당 없이" 류로 들리지만 **정반대**다. **수량당/단순곱셈**은 사면 살수록 배송비가 N씩 늘고(수량별, AP=1) — **flat/균일**은 몇 개를 사든 배송비 N원 한 번(유료, AP=blank). 구분 신호어: `flat`·`균일`·`개수 상관없이`·`수량 무관`·`주문당`·`건당` → **유료(AP 공란)**. 사용자 명시: "트위즐러는 그냥 10000원 (개당 없이 그냥 flat — 이럴 때는 AP컬럼은 blank)". `build_bulk_excel.py` 는 이런 문자열을 파싱해 유료/공란으로 채우고, product_info `bulk.shipping_type:"유료"`·`shipping_fee:N` 으로 명시하면 AP 칸을 자동으로 비운다(수량별 유형 아닐 때 부과수량 공란 처리). [[feedback_shipping_per_unit_no_bundle]]
>
> **🔑 "M개당 N원"은 액면 그대로 받는다 — "= 개당 X원" 분담 환산으로 재해석 금지 (2026-06-23 자미에슨 콜라겐 케이스):** 사용자가 **"M개당 N원"**(예: 3개당 15,000)이라고 하면 **수량별 fee=N·부과수량(AP)=M** 그대로 등록한다. 즉 **1개 주문도 N원, M개까지 N원, M+1개부터 2N원**. product_info 메모에 `"3개당 15,000원 = 개당 5,000원 (수량 분담)"` 처럼 **나눗셈 환산이 병기**돼 있어도 그건 참고 계산일 뿐 — **개당(AP=1, fee=N/M)으로 바꾸지 말 것**(개당 환산하면 1개 주문이 5,000원으로 떨어져 의도와 달라짐). 사용자 명시: "3개당 15000이라 하면 그대로 받아들여". (메인이 한 번 개당 5,000으로 해석했다가 정정받음.) [[feedback_shipping_per_unit_no_bundle]]

### 7-1. 해외 배송비 — 무게 기반 $9 + 추가kg당 $4.5 (baseline ₩10,000) (2026-06-17)

캐나다→한국 **해외 배송비는 무게 기준**으로 잡는다. 기본이 **$9부터** = 편의상 **₩10,000** 을 baseline 으로.

**산정식 (무게 기반):**
```
총무게 = 단위중량 × 수량 → kg 단위로 올림(ceil)
배송원가($) = 9 + 4.5 × (올림kg − 1)      # 첫 1kg $9, 추가 1kg당 +$4.5
배송비(KRW, 편의상 환산·라운드):
   1kg → $9     ≈ ₩10,000
   2kg → $13.5  ≈ ₩15,000
   3kg → $18    ≈ ₩20,000
```

**네이버 "수량별 부과수량(M개당)" 도출:** 무게가 1kg 경계를 넘기는 개수로 묶는다.
- 765g 제품: 1개=0.765kg→올림 1kg=$9≈₩10,000, **2개=1.53kg→올림 2kg=$13.5≈₩15,000** → 네이버 **2개당 ₩15,000**(개당 7,500).
- 814g 제품: 1개=0.814kg→1kg≈₩10,000 이지만 사용자가 개당 ₩15,000 명시 → 명시값 우선.

**적용 규칙:**
- **배송비 미지정·애매하면 → 개당 ₩10,000(1kg baseline) 디폴트.** ₩10,000 밑으로는 안 잡음(하한).
- 사용자가 명시한 배송비(개당 N원·M개당 N원·무배+흡수 등)가 있으면 그게 **최우선**. 무게식·baseline 은 미지정·하한 가드.
- product_info `shipping` 에 `"M개당 N원"` 문자열로 적으면 build_bulk_excel 이 수량별 fee=N·부과수량=M 으로 채움(개당이면 `"개당 N원"` 또는 `{per_unit_krw:N}`).

## 8. 가격표 OCR 룰 — "N/X.XX OR Y.YY EA" 양식

Loblaws / Real Canadian Superstore / Independent / No Frills 등 캐나다 슈퍼마켓 매대 가격표는 다음 양식을 자주 사용한다:

```
N/X.XX OR Y.YY EA
```

**해석:**
- **N개 묶음 X.XX** (멀티-바이 할인가)
- **단품 Y.YY EA** (단품 정상가)
- 일반적으로 **Y.YY > X.XX/N** (즉, 묶음 사면 개당 더 쌈) — 만약 단품 < 묶음 개당이면 OCR 오인일 가능성

**기본 해석 룰:**
- **단품가(Y.YY) 우선 사용** — 우리가 재구매할 때 단품 살 가능성 더 높음, 보수적 산정
- 사용자가 명시적으로 "묶음으로 산다 / 멀티-바이 적용" 하면 X.XX/N으로 변경

**검증 체크:**
- "단품가 < 묶음 개당가" 나오면 OCR 오인 의심 → 사용자에게 확인 요청
- 첫 자리 숫자(특히 1·8·5)는 사진 각도에 따라 자주 오인됨 → 사용자 정정 받기

**예시:**

| 사용자 표현 | 단품가 (사용) | 묶음 |
| --- | --- | --- |
| "2/5.50 OR 3.20 EA" | **3.20 CAD/개** | 2개 5.50 |
| "3/9.99 OR 3.79 EA" | **3.79 CAD/개** | 3개 9.99 |
| "2/15.50 OR 8.20 EA" | **8.20 CAD/개** | 2개 15.50 |

> 2026-05-17 Nestlé Snack Size Collation 케이스 학습 — 초기 OCR "2/15.50 OR 3.2 EA"로 잘못 해석해 단가 15.50으로 가정 → 사용자 정정 "2/$5.50 OR $3.20 EA, 단품 3.20 사용".
>
> 가격은 산식의 근본 입력이라 OCR 애매하면 **반드시 사용자에게 확정 받고 진행**.

## 9. 검색 유입용 / 끼워팔기 상품 — 마진 최소화 모드

사용자가 어떤 SKU를 **"검색 유입용"**, **"끼워팔기용"**, **"마진 포기/최소"** 라고 명시하면 그 상품은 traffic-driver(loss-leader)로 본다.

**적용:**
- 통상 cost-plus 마진($2~5) 대신 **마진을 최소**로 잡는다 (예: 1,000원 수준). 마진 단위가 KRW로 명시될 수 있으니 **단위(CAD vs KRW) 먼저 확인**.
- 마진 최소 모드여도 **네이버 수수료 6.6% gross-up은 그대로 적용 권장** — 안 하면 실제론 손해.
- 단, 검색 유입용은 가격 경쟁력이 핵심이라 **gross-up 적용본 / 미적용본 두 가지 가격을 모두 제시**하고 사용자가 고르게 한다.
- 한국에도 유통되는 제품이면 §0-A 국내 시세 확인은 **여전히 수행** (한국 가격과 경쟁해야 유입 효과가 생김).
- 끼워팔기 전제이므로 단품 기준 가격 열세는 **의도된 것** — §0-A "가격으로 안 됨" 판정이 나와도 자동으로 접지 말고 전략 맥락을 함께 보고.

**이유:** 객단가를 올리는 끼워팔기 상품은 그 자체 수익이 아니라 스토어 검색 노출·유입이 목적. 마진을 키우면 가격 경쟁력이 떨어져 유입 효과가 사라진다.

> 2026-05-20 Nespresso 버츄오 오다치오 캡슐 케이스에서 학습 — 한국에도 유통되는 제품, 사용자 "이건 한국에도 있어서… 검색 유입용이니 마진은 포기했어", 마진 1,000원(KRW) 명시.

## 10. 네이버 태그 — 차단 패턴 + 검색 효과 (10개 선정)

태그 10개는 **(1) 정책 차단 거르기 → (2) 검색량 검증된 키워드로 채우기** 2단계.

**🚫 차단 (태그 불가):**

## 11. 퍼센트(%) 마진 패턴 — 정액 CAD 대신 마진율 지정

기존 마진 입력은 정액(CAD/KRW $N)이었으나, 사용자가 마진을 **퍼센트(%)**로 지정하는 경우가 있다. % 마진은 해석이 갈리므로 아래 규칙으로 확정한다.

### 11-1. "수수료 감안하고 N%" — 수수료 차감 후 순마진 (net margin, 판매가 대비)

사용자가 **"수수료 감안하고 N%"** / **"수수료 빼고 N% 남겨"** / **"수수료 포함 N% 마진"** 류로 명시하면, N%는 **네이버 수수료(6.6%)를 차감한 뒤의 순마진**이며 **판매가(listed price) 대비** 비율이다. 이 경우 수수료 gross-up(÷0.934)을 **따로 적용하지 않는다** — N%가 이미 수수료 반영 후 수치이기 때문.

**산식:**
```
순마진 = 판매가 × 0.934 − 원가KRW            (실수령 − 원가)
순마진 = 판매가 × (N/100)                     (판매가 대비 N%)
→ 판매가 × 0.934 − 원가 = 판매가 × (N/100)
→ 판매가 × (0.934 − N/100) = 원가
→ 판매가 = 원가KRW ÷ (0.934 − N/100)
```
- `원가KRW = cost_original(±HST) × 환율`
- 최종 = `round_100(판매가)`
- **검산:** `판매가 × 0.934 − 원가KRW = 순이익`, `순이익 ÷ 판매가 = N%` 확인.

**예시** — Kinder Kinderini 쿠키 (2026-05-25, 원가 6.97 CAD, HST 0%, FX 1,100, 수수료 감안 순마진 30%):
- 원가KRW = 6.97 × 1,100 = 7,667 KRW
- 판매가 = 7,667 ÷ (0.934 − 0.30) = 7,667 ÷ 0.634 = 12,093.06 → round_100 → **₩12,100**
- 검산: 12,100 × 0.934 − 7,667 = 3,634.4 순이익 / 12,100 = **30.04%** ✅

> 빠른 공식: **판매가 = 원가KRW ÷ (0.934 − N/100)**. (N=30 → ÷0.634, N=25 → ÷0.684, N=20 → ÷0.734)

### 11-2. % 마진 — 수수료 관계 불명확할 때 (해석 갈림 → 반드시 명시·확인)

"마진 N%"만 있고 수수료와의 관계가 불명확하면 두 해석이 가능:
- (a) **원가 대비 마크업**: `sell_pre_fee = 원가 × (1 + N/100)`, 이후 §1 gross-up.
- (b) **판매가 대비 마진율 (pre-fee)**: `sell_pre_fee = 원가 ÷ (1 − N/100)`, 이후 §1 gross-up.

한국 이커머스 관행상 "마진 N%"는 보통 (b) 판매가 대비. 단 수수료 처리(별도 gross-up vs 마진율에 포함)가 갈리므로 **산출물에 쓴 해석을 반드시 명시하고, 애매하면 사용자에게 확인**한다. 사용자가 "수수료 감안" 류로 말하면 §11-1로 확정.

> 2026-05-25 Kinder Kinderini 케이스에서 학습 — 첫 입력 "마진 30%"가 (a)/(b)로 갈렸고, 사용자가 **"수수료 감안하고 30%"**로 확정 → §11-1 순마진 공식 적용. 첫 퍼센트 마진 케이스(기존엔 전부 정액 CAD 마진). 향후 "수수료 감안 N%" = `판매가 = 원가KRW ÷ (0.934 − N/100)`.

## 12. 세일전 정상가 책정 — 세일가 식별되면 정상가가 디폴트 (2026-06-02 격상)

**🔑 디폴트:** 매대/페이지에서 **세일가가 식별되면**(예: "SAVE $3", "new lower price", 세일가+정가 동시 표시, ~~$11.99~~ → $8.99) 사용자 명시가 없어도 **세일전 정상가를 원가로 채택**한다. (이전 룰: "세일전 가격으로" 사용자 명시가 있을 때만 → **이제 디폴트로 격상**.) 사용자가 **"세일가로 책정해"**라고 명시할 때만 세일가 사용. 세일가 식별이 안 되고 단일가만 보이면 평소대로 그 표시가 사용.

**적용 절차:**
1. 매대 세일가 무시, 사용자가 지정한 정상가 사용
2. 등록정보·product_info.json에 **정상가/세일가 둘 다 기록** (추후 추적용)
3. 세금·마진·gross-up·환산은 정상가 기준으로 그대로 진행

**기록 필드:**
- `cost_original`: 정상가 (세일전)
- `cost_pricing_basis`: "pre-sale regular price"
- `current_sale_price_cad`: 현재 매대 세일가 (참고용)
- `regular_price_cad`: 정상가 (= cost_original)
- `tax_label`: "HST 13% (regular pre-sale price — 매대 세일가 $X.XX / 정상가 $Y.YY 기준)" 식

## 13. SEO 리프레시 — 그룹상품 보존 + 전용 출력 폴더 (2026-05-31)

매일 17:00 자동 `/seo-refresh`(판매중 2개 SEO 감사·재생성) 운영 규칙. 두 가지 학습:

### 13-1. 그룹상품(`그룹상품번호` 있음)은 상품명 평탄화 금지

전체 상품 리스트(`guide/Product_*.csv`)에서 **`그룹상품번호` 컬럼이 채워진 행 = 그룹상품**(390개 중 135개). 그룹상품의 표시 상품명은 **`<그룹상품명>, <옵션1>, <옵션2>`** 형식이다.

- 예: `팀홀튼 스테인리스 스틸 트래블 텀블러 머그 650ml, 온타리오, 1개` = 그룹상품명 `팀홀튼…머그 650ml` + 색상옵션 `온타리오` + 개수옵션 `1개`.
- **그룹명(상품명)도 기존 옵션도 사후 수정 가능**(§ feedback_naver_group_lock, 2026-06-21 정정 — 이전 "기존 옵션은 잠겨 수정 불가"도 오류. 옵션도 수정 가능). 새 옵션 추가도 가능. 그룹명이 새 옵션과 안 맞으면(예: "코스트코 대용량"인데 소용량 추가) **그룹명을 일반화해 수정**한다. 콤마-옵션 나열은 **정상 구조이지 SEO 위반이 아니다.**
- 단 SEO 리프레시에선 그룹상품 **상품명을 함부로 평탄 재작성하지 않고 보존**(콤마-옵션 구조 유지), 개선은 **detail.html·태그 중심**. seo-auditor 는 콤마-옵션 형식을 위반으로 잡지 않고, 그게 유일한 지적이면 "부합" 판정. listing-writer 는 new_title=old_title. (그룹명 수정이 필요한 건 옵션 추가로 라인/용량이 달라질 때.)
- 일반(단일)상품은 종전대로 상품명도 재작성.
- **🔑 새 그룹상품을 만들 때(신규 등록)도 그룹명(상품명)에는 옵션값(색상·맛·용량)을 넣지 않는다 — 첫 번째 옵션값조차 X.** 네이버가 옵션을 별도로 결합 노출하므로 그룹명엔 옵션 빼고 제품 본명만. **§13-4의 first-option 포함 룰은 그룹이 아니라 "단일상품+옵션여럿"일 때만 적용** — 둘을 혼동 말 것. (2026-06-07 블루제이스 50주년 클린업 그룹등록에서 그룹명에 첫 옵션 '로얄블루'를 넣었다가 사용자 지적 → 제거. 같은 SKU를 일괄엑셀에서 색상별 **단일상품**으로 뺄 때는 각 상품명에 색상 포함이 정답 — 그건 단일이라 §13-4.)

> 2026-05-31 학습 — 첫 테스트에서 그룹상품 트래블머그를 `팀홀튼 트래블머그 온타리오에디션…` 으로 평탄 재작성한 게 오류. 그룹 구조·잠김을 무시한 것. `seo_pick.py` 가 `is_group`/`group_product_id` 를 후보에 실어 보내 전 단계에서 인식하도록 수정.

**새 그룹상품명 작성 — 반복 단어 + 옵션값 중복 금지 (2026-05-31 H&S 클리니컬 케이스):**

네이버는 그룹상품명과 옵션을 **합쳐서 노출**하고, 합친 문자열에 **유의어/반복 단어가 다수**면 "반복된 단어가 다수 포함" 으로 **거부**한다. 새 그룹명을 지을 때:
- 그룹명 안에서 같은 단어 반복 X (예: `두피케어 강력비듬케어` = 케어×2, 비듬×2 ❌).
- **옵션값에 든 단어를 그룹명에 또 넣지 말 것** — 옵션이 `건조케어`면 그룹명에 `케어` 넣지 말기(`두피케어`→`두피관리`로 회피). 조합 문자열(그룹명 + 각 옵션값) 전체에 중복 단어 0이 되게.
- 검증된 동의어 대체: `케어`↔`관리`(두피관리 ✅ 검색어), 카테고리어는 한 번만.

> H&S 클리니컬: `헤드앤숄더 클리니컬 비듬샴푸 두피케어 강력비듬케어` + 옵션 `건조케어` → 케어×3·비듬×2로 거부. 수정 → `헤드앤숄더 클리니컬 비듬샴푸 두피관리`(케어 0, 비듬 1) + 옵션 건조케어 = 케어×1, 반복 0. ※ 그룹명·옵션 모두 사후 수정 가능하지만 둘이 합쳐 노출되므로, 새 옵션 추가/그룹명 변경 시 조합 문자열 반복어를 점검(거부 방지).

### 13-2. SEO 리프레시 산출물은 `output/seo_refresh/` 전용 폴더

SEO 리프레시 결과물(`<slug>_seo_등록정보.md`·`_seo_detail.html`)과 `seo_refresh_log.csv` 는 **`output/seo_refresh/` 하위 폴더**에 저장한다(신규등록 산출물·모바일 동기화 파일과 분리). 이는 §3 평탄 저장 룰의 SEO 전용 예외.

### 13-3. 서로 다른 상품 묶기 → 자동 병합 금지, "분리 필요" 플래그

그룹상품이 아닌데 한 listing 에 **종류가 다른 제품**이 묶여 있으면(예: "프렌치 바닐라 카푸치노,핫 초코 믹스 454g / 500g" = 카푸치노+핫초코), SEO 리프레시가 **상품명을 하나로 합쳐 재작성하면 안 된다.** 올바른 처리는 **셀러가 별도 상품으로 분리**하는 것.

- seo-auditor: "서로 다른 상품 묶기 — 수동 분리 필요(셀러 작업)" 중대 위반으로 플래그, 합친 상품명 제안 X.

### 13-4. 단일상품 + 옵션 여럿 → 상품명에 첫 번째 옵션 포함 (2026-05-31 거버 퍼프 케이스)

네이버 스마트스토어는 **등록된 상품의 대(최상위) 카테고리를 다른 대분류로 바꿀 수 없다.** 같은 대분류 안에서의 이동(예: `패션잡화 > A > B` → `패션잡화 > C > D`)은 즉시 수정 가능하지만, 대분류 자체가 달라지면(예: `스포츠/레저 …` → `패션잡화 …`) 기존 listing에서 변경 불가.

- **해결:** 기존 제품을 **복사 → 올바른 카테고리 적용 → 새로운 상품으로 등록**(기존 잘못된 listing은 정리/판매중지). 셀러 수동 작업.
- **SEO 리프레시 시사점:** seo-auditor가 카테고리 오분류를 잡고 listing-writer가 새 카테고리를 제안할 때, 이동이 **대분류(최상위)를 넘나드는지** 반드시 표시한다.
  - 같은 대분류 내 이동 → "카테고리 수정(즉시 가능)"
  - **대분류 변경 → "⚠️ 기존 상품 수정 불가 — 복사 후 새 상품 등록 필요(셀러 작업)"** 를 Slack 요약·등록정보.md에 명시.

## 16. 일괄등록 엑셀 — 5번째 산출물 + `build_bulk_excel.py` (2026-06-05)

등록 4종(등록정보·detail.html·product_info·이미지) 만든 뒤, **네이버 스마트스토어 일괄등록 엑셀**(`<slug>_bulk_upload.xlsx`)도 함께 생성한다. 새 카피 작성이 아니라 **확정된 product_info + detail.html 을 네이버 템플릿 칸에 매핑하는 결정론적 변환** → **메인 오케스트레이터가 소유**(가격계산·저장과 같은 계열, listing-writer 아님).

**스크립트(고정):** `python3 scripts/build_bulk_excel.py <slug…> [--out 경로]`
- 입력: `output/new-item/<slug>/<slug>_product_info.json` + `_detail.html`
- **단건:** 슬러그 1개 → 그 폴더에 `<slug>_bulk_upload.xlsx` (데이터 1행).
- **배치(source-launch):** 슬러그 **여러 개를 한 번에** 넘기면 **한 엑셀에 SKU 당 한 행**으로 누적 → 기본 `output/new-item/_batch/bulk_upload_<날짜>.xlsx`(또는 `--out`). 여러 SKU 를 매번 따로 올리지 않고 **한 파일로 일괄 업로드**.
- **온디맨드 배치 (사용자 자연어 트리거):** 사용자가 **"이 제품 이 제품 묶어서 일괄(등록) 엑셀 만들어줘"**(때로 "일괄발송"이라 말해도 = 일괄**등록** 엑셀) 라고 하면 → 말한 제품들을 `output/new-item/` 의 폴더명(슬러그)으로 매칭해 `build_bulk_excel.py <slug1> <slug2> … --out output/new-item/_batch/bulk_upload_<날짜>.xlsx` 배치 실행. source-launch 크론과 별개로 **이미 등록 산출물이 있는 제품들을 임의로 골라** 한 파일로 묶는 용도. (실제 주문 출고/송장 '일괄발송'은 별개 — hanmi-flow·epost-flow 스킬.) (2026-06-05 사용자 지시)
- 공통: 1행 그룹헤더·2행 필드명 보존, 가이드 3~6행 삭제, 데이터 3행부터.
