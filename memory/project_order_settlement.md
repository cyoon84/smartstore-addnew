**매 출고일(화/금) 손익 정산 워크플로** (2026-07-08 사용자 신규 업무 추가).

**도구:** `scripts/order_settlement.py <발주발송.xlsx …>`
- 네이버 스마트스토어 **발주발송관리** 엑셀은 **암호 걸림(CDFV2)** — 기본 암호 **`1111`** (`--password`). msoffcrypto-tool 로 복호화.
- **🔑 정산 키 = 출고일(화/금), export 날짜 아님 (2026-07-09):** 출고는 매 화/금. 직전 출고 후 들어온 주문은 여러 export 파일(수·목·금아침)로 쪼개져도 **전부 다음 출고일 하나로 묶는다.** `--date` 없으면 마지막 export 날짜→그 날 포함 다음 화/금 자동계산(`ship_day()`)→`정산_<출고일>.md`. 그 출고일 export 파일들을 **전부 공백 구분으로 같이 넘기면** 상품주문번호 union-dedup 으로 합산, COGS·processed json 도 `정산_<출고일>_*.json` 하나로 누적. (2026-07-08·09 → `정산_2026-07-10.md` 하나로 통합.)
- **매출(정산 예정)** 자동 = **Σ정산예정금액(BD 컬럼)** (제품 정산=상품가−Npay수수료−매출연동수수료) **+ Σ배송비(배송비 묶음번호로 유니크)×(1−0.0363)**. 🔑 배송비엔 네이버 수수료 **3.63%만**(제품가 6.6%와 다름, §7-2). 배송비 합계(AK)는 묶음마다 행 반복이라 **묶음번호(AI)로 중복 제거** 후 합산.
- `--update-inventory <재고.xlsx>`: 주문 상품번호를 재고관리 시트 슬러그와 매칭해 **판매수량 자동 증가**(현재고 수식 차감). ⚠️ 같은 파일 두 번 돌리면 중복가산 — **출고파일당 1회만**.
- `--cogs N --shipping-cost N`: **순이익 = 매출 − 물건값 − 실배송비**.
- 출력: `output/settlement/정산_<날짜>.md` (물건값·실배송비 나중에 채워 재실행 갱신).

**🔑 추가 주문이 더 올 수 있음 — 최종 확정 전까지 열어두고 신규만 누적 (2026-07-08):** 같은 출고일에 주문이 나중에 더 들어올 수 있다. 새(전체) 발주발송 파일 다시 받으면 → 매출은 최신 전체 파일로 재계산(idempotent), 재고 판매반영은 **상품주문번호로 dedup**(정산_<date>_processed.json 에 처리 주문 기록 → 재실행·추가주문 파일에도 이중가산 X), 물건값은 추가주문분 영수증으로 `--add-cogs` 계속 누적. 실배송비·순이익은 추가주문 마감·한미 리포트 후 최종.

**3단계 입력(시점 다름):**
1. **출고일 아침** 발주발송 엑셀 → 매출(정산 예정) 자동.
2. **영수증** — 사용자가 "이건 누구누구꺼 주문이야" 하며 올리면 그 배치 **총 물건값(COGS)** 계산 → `--add-cogs`. 🔑 **영수증 반영 = COGS + Drive 업로드 + Todoist cross off, 3단계를 한 묶음으로 항상 자동 실행** (2026-07-13 확정, 사용자 "앞으로도 계속 이렇게 부탁해") — 사용자가 Drive·Todoist를 따로 언급 안 해도 매번 셋 다 처리한다:
   1. `--add-cogs` 로 그 출고일 정산 반영.
   2. Drive `receipt-qb/<출고일>/`(영수증 받은 날짜 아님, 그 정산이 속한 출고일 — §20-1)에 영수증 원본 또는 텍스트 발췌본 업로드.
   3. Todoist "사야할 제품들"(그 출고일 배치 부모, §20-2 규칙대로 기존 부모 재사용 — 새로 안 만듦) 아래 해당 항목 `complete-tasks` 로 cross off. 이미 완료돼 있으면(find-tasks 빈 결과) 스킵. 매칭은 상품명 유사.
   (2026-07-13 오레오 메이플 케이스에서 ①②만 하고 ③을 빼먹어 사용자가 "사야할 제품들도 같이 cross off해줘"로 재요청 → 이후 3단계 항상 세트로 확정.)
3. **한미 배송비 리포트** 오면 **총 실 배송비** → `--shipping-cost`. → 순이익 확정.

**🔑 손익은 주문 단위가 아니라 출고일/전체 단위로 본다 (2026-07-08 사용자):** 개별 주문 하나가 마이너스여도 연연하지 말 것 — 출고일 배치 총합(매출−물건값−실배송비)으로 판단, 하나 적자면 다른 데서 메꾼다. 저가 스낵 단품·박한 마진 SKU가 배치에 섞여도 전체가 플러스면 OK. (정산 워크플로가 이미 출고일 전체를 묶어 계산.)

**🔑 물건값(COGS)은 실현금 기준 — 글리치/무료취득 재고는 ₩0 (2026-07-08):** 출고일 순이익은 "이번 출고에 실제로 얼마 벌었나(실현금)"라, 사입재고를 쓴 주문의 COGS는 그 재고의 **실 현금 취득가**로 잡는다. **글리치·환불로 실질 무료 취득한 재고는 정산 COGS ₩0** (재고 시트의 장부값 ₩16,070은 재입고 마진용으로 유지 — 정산과 별개, 이중기준). 일반 재고는 사입단가(landed) 그대로. 영수증 사입분은 landed(상품+세금)×FX. (사용자: 애프터바이트 글리치분 → 이번 출고 실현금 ₩0.) [[feedback_costco_price_adjustment]] · LEARNED §19-3(windfall baking 금지)

**예 (2026-07-08):** 주문 7건·묶음 3개 → 매출 ₩233,448(ΣBD 214,174 + 배송정산 19,274). COGS 누적: 애프터바이트(박은영, 글리치) ₩0 + 에티튜드샴푸(박은영, Amazon CAD11.74 landed×1083) ₩12,714 + (콜라겐·밴드에이드·미스비키스×2·웨버 영수증 대기). 실배송비(한미) 대기. 재고 반영: 애프터바이트 1개 판매.

**🔑 새 발주발송 파일의 주문이 기존 Todoist에 이미 다른 배치로 있으면 ship_day() 자동계산보다 그걸 우선 (2026-07-13 검증):** `ship_day()`는 파일 export 날짜로 그 파일 전체를 한 출고일에 뭉뚱그리지만, 네이버 발주발송관리 화면은 안 나간 주문을 재노출하므로 이전에 다른 배치(다른 마감일)로 이미 Todoist에 잡아둔 주문이 새 export에도 또 찍힐 수 있다. 신규로 보이는 수취인·상품은 Todoist에서 검색해 다른 배치 부모에 이미 있는지 대조 — 있으면 그 행을 매출 계산에서 제외(수동 필터 후 재계산)하고 Todoist도 중복 추가하지 않고 사용자에게 확인만. 사용자가 배치를 확정해주면 그대로 따른다. (이수현/농심 볶음너구리 케이스 — 7/14 export에 포함돼 있었으나 이미 7/17 배치에 있어 제외 → 사용자 확인 "이수현은 7/17, 안 건드린 게 맞음".)

**🔑 배치마다 "리스팅→주문 기간"도 기록 — 매칭 실패는 "클로드 도입 이전" (2026-07-17):** 발주발송 반영 시 품목마다 등록일→주문일 경과일도 계산해 정산_<date>.md 에 남긴다. 상품명(브랜드·맛·용량)으로 `output/new-item/<slug>/<slug>_product_info.json` grep 매칭 → 등록일은 `stat -f %SB`(output/ 은 gitignore라 git 이력 없음, macOS 파일 생성시각), 주문일은 엑셀 `주문일시` 컬럼. **매칭 안 되면 "클로드 도입 이전 등록분(기록 없음)"으로 표시하고 추적 중단** — 사용자 명시 "관여한 기록 없는거면 이전에 한거야". 7/17 배치 첫 사례: 4건 매칭(도리토스 플레이밍핫나쵸 8일·메이나드 퍼지피치 28일·다우니 파라다이스680g 31일·다우니 러쉬379g 15일, 평균 20.5일), 나머지 6건은 클로드 도입 이전으로 처리.
