## 1. 페이지에 넣는 것 (매 확정마다)

1. **DB에 그 배치 제품 행 추가** (`notion-create-pages`, parent = data_source_id) → 차트 자동 누적.
2. **"출고일별 정산 요약" 표에 그 출고일 행 추가** (매출·COGS·실배송비·순이익·상태).
3. **날짜별 상세 섹션 추가** (매출/수취인별 표/COGS 목록/손익) — [[project_order_settlement]] 형식 그대로.

이미 그 출고일이 있으면 중복 추가 말고 스킵/갱신.

**🔑 날짜별 상세 섹션 = 주문별 순이익까지 한 테이블 (한미 무게별 실배송비 기준, 균등분배 금지):** 섹션 상단에 **정산 요약 표**(매출·물건값COGS·실배송비·순이익)를 별도로 두고, 이어 **수취인별 표**에 `수취인 · 제품정산 · 배송정산 · 소계 · 물건값(COGS) · 제품손익(소계−COGS) · 무게(kg) · 실배송비(한미 무게별) · 순이익(제품손익−실배송비)` 컬럼 전부. 실배송비는 한미가 준 수취인별 무게로 §7-1 무게식 `$9+$4.5×(무게−1kg)`(ceil 안 함, 연속값) ×FX 계산 → **8명(주문) 합이 한미 최종 청구 총액과 일치 검산**. 균등분배(총액÷묶음수) 쓰지 말 것 — 저가 소액주문을 적자로 왜곡(7/10 윤지애 균등분배 −₩2,798 → 무게실측 +₩1,466 반전). 무게 미제공 시에만 상품 실중량으로 무게식 추정 배분, 균등분배는 최후. 사용자: "7/14, 17일도 한미 오피셜 나오면 그렇게 만들어 계속!" — 매 출고일 반복. (LEARNED §20-11)

**페이지 레이아웃:** DB(차트) 최상단 → 출고일별 요약 표 → 날짜별 상세.

## 2. DB 스키마 · 차트 (확정형)

| 컬럼 | 타입 | 내용 |
|---|---|---|
| 제품명 | TITLE | 브랜드+라인명+타입+옵션+용량 (§3 규칙) |
| 개수 | NUMBER | **주문 개수** (금액 아님, §4 규칙) |
| 출고일 | DATE | 그 배치 출고일 |
| 월 | RICH_TEXT | `2026-07` — 차트 stack 축 |

**차트 = "제품별 주문수" 탭 하나뿐.** GROUP BY 제품명 · AGGREGATE count · 월 컬럼으로 stack.

**🔑 Notion 무료 플랜 = chart view 1개 제한.** 월별로 DB/차트를 새로 만들지 말고 **이 DB 하나에 모든 달을 계속 누적**(그래서 제목에 월 표기 없음 — 월은 `월` 컬럼으로 구분). **새 chart view 를 `create-view` 로 추가 시도 금지** — 한도 걸리고 사용자가 UI에서 지워야 함.

**차트 = 금액이 아니라 주문 빈도.** COGS SUM 기준으로 시도했다가 "Something is wrong with your chart data" 렌더 실패 + 사용자 정정("제품별 주문수로 해야할거같은데"). 금액은 정산 페이지 본문 COGS 섹션에 이미 있으니 DB에 중복 보관 안 함.

## 3. 제품명 규칙 — 축약 금지

정산 md 자체가 축약형이어도 **DB엔 정확한 풀네임**을 넣는다 (사용자: "이렇게 봐선 정확히 뭔지 모름").

- ① **옵션/변형(맛·향·색상) 포함**, 서로 다른 옵션은 **행 분리** — `미스비키스 감자칩 - 올드레스드` / `미스비키스 감자칩 - 할라피뇨` (한 행에 "올드레스드+할라피뇨" ❌)
- ② **브랜드+라인명+타입 다 포함, 임의 축약 금지** — `타이드 세탁캡슐 63입` ❌ → `타이드 파워포드 스포츠 세탁캡슐 63입` ✅
- ③ **등록 산출물 있으면 확정 명칭 사용** — 용량(g/ml)으로 `output/new-item/<slug>_<용량>/*_product_info.json` grep → `title_ko` 채택. `다우니 퓨전 598g` ❌ → `다우니 언스토퍼블 퓨전 트와일라잇자스민 598g` ✅
- ④ 내부용 메모는 빼기 — `애프터바이트(무료취득)` ❌ → `애프터바이트` ✅ (글리치·무료취득은 정산 본문에만)

## 4. 🔑 `개수` = "몇 개 주문했나" — 멀티팩 기획상품은 1

정산 md 의 `×2`·`2개`·`2병` 표기를 그대로 수량 2로 넣지 말 것. **묶음/세트로 파는 SKU 는 그 세트를 1개 주문한 것 = 개수 1**, 세트 구성은 **제품명에 표기**.

- `세라비 크림 453ml 2개입 세트` → 개수 **1** (2개입 세트 1개 주문)
- `웨버 트리플액션 조인트케어 (2개 묶음세트)` → 개수 **1** (원가가 `$29.99×2`로 보여도 2병 묶음 기획가 — **가격 산식의 ×2 를 주문수량으로 오독 금지**)
- 서로 다른 맛/옵션을 각각 산 경우만 **행 분리**(각 1)

애매하면 추측 말고 사용자에게 확인.

## 5. Notion MCP 실전 노하우 (시행착오 정리)

- **차트 DSL** (통과하는 유일 문법): `CHART bar AGGREGATE "count" GROUP BY "제품명"` — **세미콜론 없이 공백으로 이어 쓰고**, AGGREGATE 뒤엔 **함수명을 소문자 따옴표**(`"count"`/`"sum"`, 프로퍼티명 아님). `SORT BY` 는 quoted 프로퍼티만 허용 — count 자체로는 정렬 불가.
- **차트를 페이지에 심는 법:** `?v=<view-id>` 쿼리스트링은 실패(Notion 이 base uuid 같은 `<database>` 태그를 동일 블록으로 병합/dedup → `inline="false"` 링크로 collapse). **DB 자체를 `is_inline: true`**(`notion-update-data-source`) 로 두고 `<database inline="true">` 블록을 원하는 위치에 배치하는 게 정답. (`create-view(parent_page_id=…)` 로 만들면 페이지 끝에 붙는데, 그 url 은 **페이지를 다시 fetch 해서 새로 붙은 블록의 url 을 읽어야** 함 — 응답의 `view://<uuid>` 를 dash 제거해 쓰면 틀림.)
- **`replace_content` 로 페이지 재구성 시** 유지할 하위 DB/페이지를 새 content 에서 빠뜨리면 "하위 페이지/DB 삭제" 에러 → 유지할 건 `<database url="…" data-source-url="…">이름</database>` 태그로 포함, 진짜 지울 거면 `allow_deleting_content: true`.
- **컬럼 리네임:** `notion-update-data-source` statements `RENAME COLUMN "A" TO "B"`. 차트 config 가 GROUP BY 제품명 + count 뿐이라 숫자 컬럼 리네임은 차트를 안 깨뜨림.
- **폐기된 접근:** SVG 이미지 첨부(`notion-create-attachment` + `<image>`) 방식은 사용자 요청("svg말고 노션 차트로")으로 폐기, 파이차트도 미채택. 네이티브 DB+chart view 만 사용.

## 6. 🔑 매월 1일(이후 첫 세션) = 전달 월간 누적 정산도 같은 페이지에 추가 (2026-07-17)
